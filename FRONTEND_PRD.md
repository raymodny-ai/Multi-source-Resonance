## 1. 项目背景与目标

### 1.1 项目背景

多源共振监控系统（Multi-source Resonance）是一套面向美股量化交易的全自动监控后端，已实现 **做市商 GEX Gamma 敞口计算、VIX 期限结构分析、加密衍生品去杠杆监控、机构暗盘三驾马车验证（DIX / Short Volume / Stockgrid 净头寸）、多维共振信号评分（Level 1-3）与触发状态机、Hawkes 过程自激抛售测算、多渠道告警推送（Email / Telegram / Discord）、以及基于 APScheduler 的盘中与盘后定时调度引擎** 等完整后端能力。

当前系统以 **无头终端模式** 运行，所有监控数据、信号触发、告警通知仅通过日志文件和消息推送输出。对于量化交易员和风控人员而言，缺乏一个**直观、可交互、支持实时刷新**的前端 Web UI 来承载以下核心需求：

- 在开盘期间快速扫视四大维度状态（一眼看清是否需要行动）
- 追踪暗盘指标的时序变化趋势（DIX 走势、净头寸背离标注）
- 回溯历史信号时间轴（复盘当周共振触发节奏）
- 实时感知系统健康度（爬虫连通性、降级模式、任务执行日志）
- 在页面上直接调整信号阈值和通知配置（无需 SSH 改 `.env` 重启）

### 1.2 产品目标

| 目标 | 描述 | 衡量标准 |
|------|------|---------|
| **可视化总览** | 用户打开页面 3 秒内获取四大维度当前状态 | 仪表盘加载时间 < 3s |
| **深度探索** | 可下钻查看任一维度历史趋势与明细 | 图表交互响应 < 500ms |
| **信号可回溯** | 完整时间轴展示所有历史共振触发 | 支持 90 天历史查询 |
| **系统可观测** | 一目了然爬虫健康度、降级模式、调度日志 | 状态刷新延迟 < 15s |
| **配置即页面** | 阈值/频率/通知在页面上修改即生效 | 配置变更 < 5s 生效 |

### 1.3 前端定位

前端 UI 是**纯展示与轻量配置层**，不承载核心计算逻辑。所有量化计算（GEX、DIX、Hawkes 等）由现有 Python 后端完成，前端通过 RESTful API 读取聚合数据并渲染。

```
┌──────────────────┐     REST API      ┌──────────────────┐
│   React SPA      │ ◄───────────────► │  FastAPI Middle   │
│   (Port 3000)    │   JSON over HTTP  │  (Port 8000)     │
└──────────────────┘                   └────────┬─────────┘
                                                │
                                         ┌──────▼─────────┐
                                         │  Existing Core  │
                                         │  (Python 3.10+) │
                                         │  - main_scheduler│
                                         │  - quant_logic  │
                                         │  - signal_engine│
                                         │  - SQLite DB    │
                                         └─────────────────┘
```

## 2. 用户画像与使用场景

### 2.1 用户画像

#### 用户 A：独立量化交易员 (Primary)
- **背景**: 自主管理资金，使用多因子模型指导交易决策
- **技能**: 熟悉 GEX/DIX 概念，能读懂期权链数据
- **行为**: 开盘期间每 15-30 分钟扫一眼仪表盘，收盘后深度复盘
- **痛点**: 需要快速确认是否触发"右侧做多"条件，不想盯 4 个数据源
- **设备**: 桌面为主（多屏幕），偶尔手机查看

#### 用户 B：机构风控分析师 (Secondary)
- **背景**: 负责监控组合尾部风险，需全天候关注市场微观结构异常
- **技能**: 精通波动率曲面、期限结构，关注系统性风险信号
- **行为**: 收到 LEVEL 3 告警后，打开 Web UI 核实各维度详情
- **痛点**: 通知收到后需要快速验证信号质量（是否假阳性）
- **设备**: 桌面端，可能需要投屏到大屏

#### 用户 C：系统运维人员 (Tertiary)
- **背景**: 负责维护监控系统正常运行
- **技能**: DevOps，熟悉 Python/Linux
- **行为**: 每天检查一次系统状态页，排查爬虫失败原因
- **痛点**: 不想 SSH 进服务器看日志，希望在页面上看到降级状态
- **设备**: 桌面端

### 2.2 核心使用场景

| 场景 ID | 场景名称 | 用户 | 频率 | 关键页面 |
|---------|---------|------|------|---------|
| S1 | 盘中快速扫视 | A | 每 15-30 分钟 | 实时仪表盘 |
| S2 | 暗盘深度分析 | A/B | 每日盘后 | 暗盘详情页 |
| S3 | 信号核实与决策 | A/B | 收到告警后 | 共振信号面板 |
| S4 | 历史复盘 | A | 周末 | 信号时间轴 |
| S5 | 爬虫故障排查 | C | 每日/故障时 | 系统状态监控页 |
| S6 | 阈值调优 | A/B | 月度 | 参数配置后台 |
| S7 | 手机快速确认 | A/B | 收到 Telegram 后 | 移动端仪表盘 |

#### 3.1.2 四维度信息卡

每张卡片包含以下元素：

| 元素 | 说明 | 数据来源 |
|------|------|---------|
| **得分进度条** | 环形/线形进度条，当前分/满分 (如 1.5/1.5) | API: `GET /api/dashboard/scores` |
| **状态指示灯** | 🟢/🟡/🔴 三色，对应 POSITIVE/CONVERGING/NEGATIVE | 同上 |
| **一句话摘要** | 关键变化描述（如 "GEX 已翻正至 +$150M"） | 同上 |
| **迷你趋势图** | 最近 N 个数据点的缩略折线图（点击下钻） | API: `GET /api/gex/history?limit=24` |
| **最后更新时间** | 数据源最后刷新的绝对时间 | 同上 |

**GEX 卡片**:
- 本地估算 GEX / 校准后 GEX 双值显示
- Flip Zone 上下界标注
- Put Wall 价位（灰色参考线）
- α 校准系数小字标注

**VIX 卡片**:
- VIX 现货 / VX1 / VX2 三值显示
- 期限结构比值 (VX1/VX2)
- 恐慌溢价百分比
- Contango/Backwardation 状态标签

**Crypto 卡片**:
- BTC 资金费率 (实时百分比)
- OI 持仓量 (美元) + 1h 变化率
- OI 暴跌 / 费率异常 / 杠杆清洗 三个 Flag
- ELR 预估杠杆率

**Darkpool 卡片**:
- DIX 百分比值
- 卖空比百分比值
- Stockgrid 20d / 60d 斜率
- 底背离 Flag + Golden Cross Flag
- DBMF 均线收复 Flag
- 可用源计数 (如 3/3, 2/3 DEGRADED)

#### 3.1.3 共振得分条

位于四张卡片下方的横向通栏：

- **大号数字**: 共振总分 / 满分 (如 4.8 / 5.0)
- **百分比**: (96%)
- **预警等级**: LEVEL_3 / LEVEL_2 / LEVEL_1 / NO_SIGNAL
- **等级颜色**: LEVEL 3 = 红色脉冲动画, LEVEL 2 = 橙色, LEVEL 1 = 黄色, NO_SIGNAL = 灰色
- **触发条件列表**: 4 个 icon 点亮/熄灭
- **Hawkes 分支比**: 小字标注 + 状态 (SUBCRITICAL / CRITICAL / SUPERCRITICAL)

#### 3.1.4 最近告警列表

仪表盘底部显示最近 5 条告警记录的简要列表：

| 时间 | 等级 | 得分 | 触发条件摘要 | 状态 |
|------|------|------|-------------|------|
| 14:15 | LEVEL 3 | 4.8 | GEX+VIX+Crypto+Darkpool | 🔴 未确认 |
| 11:30 | LEVEL 1 | 2.5 | GEX+VIX | ✅ 已确认 |
| 昨日 | LEVEL 2 | 3.2 | VIX+Darkpool | ✅ 已确认 |

点击行跳转到 [共振信号面板](#33-共振信号面板-p0)。

#### 3.2.2 图表详细规格

**DIX 趋势图**:
- **类型**: 折线图 (ECharts / Recharts LineChart)
- **X 轴**: 日期 (90 天范围)
- **Y 轴**: DIX % (0-100)
- **参考线**: 45% 阈值水平虚线 (红色, `strokeDasharray`)
- **数据点**: 超过阈值时圆点高亮红色
- **Tooltip**: 日期 + DIX 值 + 是否触发信号
- **交互**: 支持缩放 (brush select) 和拖拽平移

**Short Volume 趋势图**:
- 与 DIX 图相同规格，Y 轴为卖空比 %
- 45% 阈值参考线

**Stockgrid 净头寸趋势图**:
- **类型**: 双折线图
- **Line 1**: 20 日斜率序列 (绿色), Line 2: 60 日斜率序列 (蓝色)
- **标注点**: 
  - **底背离标注**: 红色向上三角 (价格跌 + 20d 斜率升)
  - **Golden Cross 标注**: 金色星形 (20d > 0 AND 60d > 0)
- **零轴参考线**: 水平虚线
- **X 轴**: 日期 (120 天)
- **Tooltip**: 日期 + 20d 值 + 60d 值 + 价格趋势 + Flag 状态

#### 3.4.4 事件聚合 (Incident) 模式 (P1)

**问题**: 市场剧烈震荡时，同一共振事件可能在 2 小时内反复在 Level 2 和 Level 3 之间震荡触发，平铺告警列表会被密集短时触发记录淹没，导致复盘困难。

**方案**: 将平铺的"告警列表"升级为**事件聚合模式**。

**聚合规则**:

| 规则 | 描述 |
|------|------|
| **时间窗口** | 2 小时滑动窗口 |
| **聚合条件** | 同一交易日内 Level 2 及以上等级的触发记录 |
| **折叠逻辑** | 窗口内首次触发生成"母事件 (Incident)"，后续触发合并为该 Incident 的子记录 |
| **展开/折叠** | 列表默认显示折叠的 Incident 卡片，点击展开查看历次触发明细 |
| **状态提升** | 子记录中若出现 Level 3，母事件状态升级为 Level 3 |

**Incident 卡片布局**:

```
┌──────────────────────────────────────────────────────────┐
│ 🔴 INCIDENT #14    2026-06-09 盘中异常抛售事件             │
│ 持续时间: 11:30 → 14:15 (2h45m)                           │
│ 最高等级: LEVEL 3  最高得分: 4.8/5.0  触发次数: 7 次       │
│ ┌────────────────────────────────────────────────┐       │
│ │ 子触发明细 (7次)                          [展开] │       │
│ │ 14:15  LEVEL 3  4.8  GEX+VIX+Crypto+Darkpool │       │
│ │ 13:45  LEVEL 2  3.2  VIX+Darkpool            │       │
│ │ 13:00  LEVEL 2  3.5  GEX+VIX+Darkpool        │       │
│ │ 12:15  LEVEL 2  3.1  GEX+Darkpool            │       │
│ │ 11:30  LEVEL 2  3.3  VIX+Darkpool            │       │
│ └────────────────────────────────────────────────┘       │
│ [标记Incident为已复盘] [导出Incident报告]                  │
└──────────────────────────────────────────────────────────┘
```

**后端支持**:

- 新增 `incidents` 表和 `/api/incidents` 端点
- 信号触发时自动关联到当前活跃的 Incident 或创建新的 Incident
- Incident 自动关闭条件: 连续 2 小时无新触发

### 4.5 时区处理策略 (P0)

**问题**: 系统后端数据严格依赖美股交易时段 (EST/EDT)，但前端团队可能在 UTC+8 等亚洲时区开发或使用，极易产生时间轴错位。

**强制规范**:

| 层级 | 规范 | 说明 |
|------|------|------|
| **后端 API 传输** | 全部时间戳使用 **UTC** 或严格 **ISO 8601 带时区偏移** (如 `2026-06-09T14:15:00-04:00`) | 禁止使用不含时区的本地时间字符串 |
| **数据库存储** | 统一 UTC 存储 (`DATETIME` 字段插入时转换为 UTC) | 现有 DB 需检查 EST 时间戳并修正 |
| **前端接收** | 解析 ISO 8601 为 `dayjs` 对象 | 使用 `dayjs.utc()` 统一解析 |
| **前端展示** | 全部展示由前端根据用户选择的时区完成格式化 | 后端绝不返回已格式化的本地时间字符串 |

**全局时区切换器**: 详见 [§6.6 全局时区切换器](#66-全局时区切换器)。

**验收标准**:

| AC-ID | 验收条件 |
|-------|---------|
| AC-TZ-01 | 后端所有 API 响应中时间字段均为 ISO 8601 带时区偏移格式 |
| AC-TZ-02 | 数据库中所有 `DATETIME` 字段统一为 UTC |
| AC-TZ-03 | 切换时区后，所有页面时间显示（仪表盘、图表 X 轴、告警列表）在 500ms 内统一刷新 |
| AC-TZ-04 | 默认时区锁定为 `America/New_York` (EST/EDT 自动夏令时切换) |

### 4.6 数据陈旧度 (Staleness) 处理 (P0)

**问题**: 网络波动导致 WebSocket/SSE 断开时，如果前端没有任何降级提示，量化交易员可能根据已经卡死或陈旧的数据做出错误决策。

**陈旧度判定规则**:

| 状态 | 条件 | 视觉处理 |
|------|------|---------|
| **实时 (FRESH)** | 最近一次更新 < 30 秒 | 正常显示，绿色小圆点 `● LIVE` |
| **稍旧 (STALE_WARN)** | 30 秒 ≤ 最后一次更新 < 2 分钟 | 卡片出现淡黄色边框，数值旁显示 "30s 前" |
| **陈旧 (STALE)** | 2 分钟 ≤ 最后一次更新 < 5 分钟 | **卡片整体变灰** (opacity: 0.5)，叠加斜线水印样式 (hash pattern)，显示 "上次更新: X 分钟前" |
| **断开 (DISCONNECTED)** | 最后一次更新 ≥ 5 分钟 或 WebSocket 显式断开 | 卡片完全置灰 + 🔴 "数据中断" 标签 + 顶部通栏红色警告条 "⚠️ 实时数据连接已中断 X 分钟，请检查网络" |

**UI 实现要点**:

- 每个实时数据卡片独立追踪 `lastUpdatedAt` 时间戳
- 使用 `useStaleness(lastUpdated)` 自定义 Hook 统一计算陈旧状态
- 斜线水印 (hash pattern) 通过 CSS `repeating-linear-gradient` 实现
- 仪表盘顶部全局状态栏：正常时显示 "● 实时连接"，中断时显示 "⚠️ 数据连接中断 [重试]"

**验收标准**:

| AC-ID | 验收条件 |
|-------|---------|
| AC-SL-01 | 数据超过 2 分钟未更新时卡片自动变灰 (不依赖用户刷新) |
| AC-SL-02 | 斜线水印清晰可见，但不过度遮挡数值 |
| AC-SL-03 | 数据恢复更新后 3 秒内卡片恢复正常色彩 |
| AC-SL-04 | 首次加载时卡片显示骨架屏，不作为"陈旧"处理 |

### 6.7 数据陈旧视觉置灰规范 (P0)

**设计原则**: 严禁向交易员展示具有误导性的过期"健康"状态。

**视觉分级**:

| 陈旧等级 | CSS 效果 | 图示说明 |
|---------|---------|---------|
| FRESH (< 30s) | 无任何效果 + 绿色 `● LIVE` 角标 | 正常色 |
| STALE_WARN (30s-2min) | `border: 1px solid #eab308` 淡黄边框 + `opacity: 0.85` | 轻微降饱和度 |
| STALE (2min-5min) | `opacity: 0.45` + `background: var(--stale-overlay)` 斜线水印 + `filter: grayscale(0.6)` | 显著灰化+斜线 |
| DISCONNECTED (≥5min) | `opacity: 0.3` + `filter: grayscale(1)` + 顶部红色通栏 `⚠️ 实时数据连接已中断 X 分钟` | 完全灰化+红色告警条 |

**斜线水印 CSS 实现**:

```css
.stale-card {
  opacity: 0.45;
  filter: grayscale(0.6);
  background: repeating-linear-gradient(
    -45deg,
    transparent,
    transparent 6px,

    rgba(148, 163, 184, 0.15) 12px

}

  position: absolute;
  top: 8px;
  right: 12px;
  font-size: 11px;

}
```

**实施**: 所有实时数据组件共享 `useStaleness(lastUpdatedAt)` Hook，返回 `{ level, label, className }` 三元组，组件据此添加 CSS class。

### 6.8 移动端图表优雅降级 — Sparklines (P1)

**问题**: 原方案"静态图片"后端渲染截图成本高，"纯数字"丢失趋势信息。

**方案**: 采用 **Sparklines（微缩图）** 方案。

**Sparkline 规格**:

| 属性 | 值 |
|------|-----|
| 宽度 | 卡片内容区 100% (约 280px) |
| 高度 | 40px |
| 数据范围 | 最近 7 个交易日 |
| 坐标轴 | 无 (纯折线) |
| 颜色 | 正值趋势绿色，负值趋势红色 |
| 端点标记 | 首尾各一个 4px 圆点 |

**示例**:

```
┌─────────────────────────┐
│ DIX 暗盘指标      47.2% │
│ ╱╲   ╱╲                │  ← Sparkline (40px 高)
│ ╱  ╲_╱  ╲___           │
│ [横置手机查看完整图表]    │
└─────────────────────────┘
```

**横屏模式 (Landscape Mode)**:

- 竖屏时卡片底部显示提示文字: `↻ 横置手机查看完整交互式图表`
- 检测到 `orientation: landscape` 时，自动展开完整的 ECharts 图表
- 利用手机原生屏幕的物理宽度 (约 800px) 解决图表拥挤问题

**技术实现**:

- 使用 `recharts` 的 `<LineChart>` 组件，关闭所有坐标轴和网格线
- 响应 `window.matchMedia('(orientation: landscape)')` 切换视图
- 横屏全屏模式下隐藏浏览器 Chrome，提供沉浸式图表体验

**验收标准**:

| AC-ID | 验收条件 |
|-------|---------|
| AC-SP-01 | 竖屏 (< 768px) 时显示 Sparkline，高度不超过 50px |
| AC-SP-02 | Sparkline 颜色随趋势方向变化 (正值绿/负值红) |
| AC-SP-03 | 横屏时自动切换为完整 ECharts 图表，纵屏时切回 Sparkline |
| AC-SP-04 | 竖屏时明显提示"横置手机查看完整图表" |

### 10.2 数据库为前端新增的查询

现有 4 张核心表已满足前端查询需求，无需新增表。以下是前端需要的关键查询映射：

| 页面 | 查询表 | 关键字段 |
|------|--------|---------|
| 仪表盘 GEX 卡 | `gex_history` | gex_local, gex_calibrated, put_wall_level, flip_zone_* |
| 仪表盘暗盘卡 | `dark_pool_metrics` | dix_value, chartexchange_short_ratio, stockgrid_*_slope |
| 仪表盘 Crypto 卡 | `crypto_derivatives` | btc_funding_rate, btc_oi, oi_crash |
| GEX 历史图 | `gex_history` | timestamp, gex_local, gex_calibrated |
| 暗盘历史图 | `dark_pool_metrics` | date, dix_value, chartexchange_short_ratio, stockgrid_*_slope |
| 信号时间轴 | `signal_alerts` | trigger_time, total_score, alert_level, acknowledged |
| 告警列表 | `signal_alerts` | 全部字段 |

VIX 数据需要新增 `vix_history` 表或在现有表基础上补充查询。建议在 Phase 8 中新增：

```sql
CREATE TABLE IF NOT EXISTS vix_history (
    timestamp DATETIME PRIMARY KEY,
    vix_spot REAL,
    vx1 REAL,
    vx2 REAL,
    term_structure_ratio REAL,
    term_structure_state TEXT,
    panic_premium REAL
);
```

### 10.3 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| SQLite 并发读性能瓶颈 | 仪表盘频繁刷新时响应慢 | WAL 模式已启用；前端使用 TanStack Query 缓存，5 分钟内不重复请求 |
| ChartExchange API 不稳定 | 暗盘详情页数据缺失 | 降级模式标识清晰展示，图表中缺失数据留空而非报错 |
| 前端打包体积过大 | 移动端加载慢 | ECharts 按需引入 (Tree Shaking)；路由懒加载 |
| WebSocket 连接断开 | 日志流中断 | 自动重连机制 (指数退避)；断线时显示 "连接断开" 提示 |
