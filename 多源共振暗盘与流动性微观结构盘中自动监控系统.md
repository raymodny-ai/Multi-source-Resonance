# 多源共振暗盘与流动性微观结构盘中自动监控系统

## 1. 项目概述

### 1.1 项目定位

本系统基于 **WebSocket + EventBus 实时事件驱动（Push）架构**，实现多维度金融数据的实时采集、量化分析和信号触发。系统通过追踪美股暗盘机构资金动向、做市商 Gamma 敞口、VIX 波动率期限结构和加密市场杠杆清洗，进行四维度共振评分，自动识别"流动性清算衰竭"级别的抄底信号。

**当前架构版本: v2 (Push 实时流)** — 全面替代基于 APScheduler 的定时轮询（Pull）模式。

### 1.2 核心能力

| 能力 | 实现方式 |
|------|---------|
| 实时数据流 | Hyperliquid DEX WebSocket 长连接 → EventBus 即时推送 |
| 四维共振评分 | GEX + VIX + Crypto + Darkpool，满分 5.0，阈值 3.5 |
| 多渠道告警 | Email (SMTP) + Telegram Bot + Discord Webhook |
| 降级容错 | Hyperliquid → CCData REST；yfinance → FINRA 管道文件 |
| 异常检测 | Hawkes Process 自激抛售分支比、资金费率异常、OI 断崖检测 |
| 状态持久化 | SQLite WAL 模式，告警冷却期 30 分钟防重复 |

### 2.2 StreamEngine 启动序列

```python
# main_stream.py — 系统入口
from data_stream.stream_engine import StreamEngine

engine = StreamEngine()
engine.start()
```

启动顺序（`stream_engine.py`）：
1. `EventBus.start()` — 启动事件分发器
2. `SignalPipeline.start()` — 订阅所有 Topic
3. `RESTPollScheduler.start()` — 启动 4 个轮询任务（GEX/VIX/AXLFI/DBMF）
4. `HyperliquidStream.start()` — 建立 WebSocket 长连接

## 4. 四维度共振评分矩阵

### 4.1 评分体系

系统采用分值累计制，满分 5.0 分，触发门槛为 3.5 分（LEVEL_3）。

| 监测维度 | 满分 | 左侧枯竭区条件 | 右侧确认反转条件 |
|---------|------|---------------|-----------------|
| **1. Gamma 状态** | 1.5 分 | GEX 负值大幅收敛 | GEX 完全跨越翻转线 (>0) |
| **2. VIX 期限结构** | 1.0 分 | VX1/VX2 > 1.15（极度 Backwardation） | VX1/VX2 < 1.0 且斜率向下 |
| **3. 加密杠杆清洗** | 1.0 分 | OI 断崖下跌 >15% | 费率转正 + 去杠杆完成 |
| **4. 暗盘吸筹** | 1.5 分 | 三选二聚合信号 | DBMF 收复 5 日均线 |

### 4.2 预警级别

| 级别 | 分数区间 | 含义 | 通知渠道 |
|------|---------|------|---------|
| **LEVEL_3** | ≥ 3.5 | 全维度共振，最高级抄底信号 | Email + Telegram + Discord |
| **LEVEL_2** | ≥ 3.0 | 多维度共振，密切监控 | Email + Discord |
| **LEVEL_1** | ≥ 2.0 | 初步关注信号 | Email |
| **NO_SIGNAL** | < 2.0 | 无信号 | — |

### 4.3 暗盘降级评分

当某些暗盘数据源不可用时，自动放弃该校验，将判定权交给其他存活源，权重不作调减：
- 3 源全部存活 → 需要 2 个触发
- 2 源存活 → 需要 2 个触发（印证更严格）
- 1 源存活 → 需要 1 个触发
- 0 源存活 → 得分降为 0，发送 CRITICAL 告警

## 5. 核心模块说明

### 5.1 EventBus — 全局事件总线

`data_stream/event_bus.py`（265 行）

基于 `asyncio.Queue` 的异步 Pub/Sub 模式，支持 10 个预定义 Topic：

| Topic 常量 | 含义 | 数据负载 |
|-----------|------|---------|
| `crypto.funding_rate` | 资金费率更新 | `{"rate": float, "coin": str, "timestamp": datetime}` |
| `crypto.open_interest` | 持仓量更新 | `{"oi": float, "oi_base": float, "mark_price": float}` |
| `gex.update` | GEX/DIX 更新 | `{"gex": float, "dix": float, "price": float}` |
| `vix.term_structure` | VIX 期限结构 | `{"spot": float, "vx1": float, "vx2": float}` |
| `darkpool.axlfi` | AXLFI 暗盘 | `{"dollar_dp_position": [], "divergence": bool, ...}` |
| `short_volume.spy` | SPY 做空数据 | `{"ratio": float, "source": str}` |
| `dbmf.recovery` | DBMF 收复 | `{"recovery": bool, "price": float}` |
| `data.source_error` | 数据源异常 | `{"source": str, "error": str}` |
| `data.source_recovered` | 数据源恢复 | `{"source": str}` |
| `system.shutdown` | 系统关闭 | — |

### 5.2 HyperliquidStream — WebSocket 连接器

`data_stream/hyperliquid_stream.py`（337 行）

- **连接**: `wss://api.hyperliquid.xyz/ws`
- **订阅消息**: `{"method": "subscribe", "subscription": {"type": "activeAssetData", "coin": "BTC"}}`
- **推送字段解析**: `funding` → `crypto.funding_rate`, `openInterest` × `markPx` → `crypto.open_interest`
- **自动重连**: 指数退避 1s → 2s → 4s → ... → 60s (上限)
- **Ping/Pong 保活**: 每 30s 发 ping，10s 超时断开
- **降级触发**: 连接断开 → `data.source_error` 事件 → SignalPipeline 启动 CCData REST 轮询

### 5.3 SignalPipeline — 事件驱动信号管线

`data_stream/signal_pipeline.py`（679 行，核心模块）

订阅全部 8 个业务 Topic，数据到达即时触发流程：

1. **维度评估** — `_evaluate_crypto_dimension()` / `_evaluate_gex_dimension()` / `_evaluate_vix_dimension()` / `_evaluate_darkpool_dimension()`
2. **就绪检查** — `_check_all_dimensions_ready()`，四维全部就绪时触发共振评分
3. **防抖** — 共振评分最小间隔 `EVAL_COOLDOWN_SECONDS = 30` 秒
4. **共振评分** — `_evaluate_resonance()`，从 DB 读最新数据，计算四维分数
5. **告警推送** — `_send_alert()`，格式化 → 写 DB → 多渠道发送

### 5.4 RESTPollScheduler — 轻量轮询调度器

`data_stream/rest_poll_scheduler.py`（468 行）

替代 APScheduler，使用 `asyncio.create_task` + `asyncio.sleep` 实现：
- 4 个独立异步轮询任务：`_poll_gex_dix()`, `_poll_vix()`, `_poll_axlfi()`, `_poll_dbmf()`
- 盘后做空数据任务：`run_afterhours_short_volume()`（yfinance → FINRA 降级）
- 自动识别美股交易时段（美东 9:30-16:00）切换间隔
- 所有数据发布到 EventBus（非直接写 DB）

### 5.5 量化逻辑层

| 模块 | 文件 | 功能 |
|------|------|------|
| GEXCalculator | `quant_logic/gex_calculator.py` | GEX 估算、α 系数校准 |
| VIXAnalyzer | `quant_logic/vix_analyzer.py` | Contango/Backwardation 判定、恐慌溢价 |
| CryptoLeverageCleaner | `quant_logic/crypto_leverage_cleaner.py` | OI 暴跌检测、费率异常、去杠杆确认 |
| DarkPoolVerifier | `quant_logic/darkpool_verifier.py` | 底背离验证、信号确认 |

### 5.6 告警推送

`notification/alert_sender.py`（400 行）

支持三种渠道并发推送：

```python
from notification.alert_sender import AlertSender

sender = AlertSender()
sender.send_multi_channel_alert(
    subject="[LEVEL_3] 共振抄底信号触发",
    message=formatted_alert,
    channels=['email', 'telegram', 'discord']
)
```

LEVEL_3 告警消息模板遵循 PRD 4.2 节规范，包含四维度详情、触发条件列表和 Hawkes Process 测算结果。

SQLite WAL 模式，位于 `database/monitoring.db`。

| 表 | 字段 |

| `dark_pool_metrics` | date, dix_value, chartexchange_short_ratio, stockgrid divergences, dbmf_ma5_recovery, signal flags |
| `crypto_derivatives` | timestamp, btc_funding_rate, btc_oi, oi_change_1h, liquidation_spike, funding_anomaly, oi_crash, leverage_cleanup |
| `signal_alerts` | id, trigger_time, total_score, 各维度得分, alert_level, hawkes_branching_ratio, details (JSON) |

## 7. 异常容错与降级

### 7.1 数据源降级

| 场景 | 行为 |
|------|------|
| Hyperliquid WS 断连 | 自动重连（指数退避）；降级到 CCData REST 轮询 |
| yfinance 做空数据失败 | 降级到 FINRA 管道文件解析 |
| SqueezeMetrics CSV 不可用 | GEX/DIX 维度跳过，其他三维继续评分 |
| AXLFI 暗盘失败 | 暗盘维度退化为仅依赖做空数据和 DBMF |
| 全部暗盘源失效 | 暗盘得分降为 0，发出 CRITICAL 告警 |

### 7.2 重试策略

- 所有 HTTP 请求封装 `tenacity`，指数退避重试（2s → 30s，最多 3 次）
- WebSocket 自动重连，延迟 1s → 60s 上限

## 9. 技术架构演进

| 版本 | 架构模式 | 数据获取 | 调度器 | 入口文件 |
|------|---------|---------|--------|---------|
| **v1** (Phase 1-7) | Pull 定时轮询 | REST 拉取 | APScheduler cron | `main_scheduler.py` |
| **v2** (Current) | Push 实时事件驱动 | WebSocket + EventBus | asyncio.create_task | **`main_stream.py`** |

v1 → v2 核心变更：
- FMP 短卖 JSON → yfinance 做空数据（免费, 无需 API Key）
- CCXT/Coinglass 加密数据 → Hyperliquid DEX WebSocket（去中心化, 免费）
- ChartExchange/Stockgrid 暗盘 → AXLFI 公开 API
- APScheduler 固定时刻拉取 → EventBus 事件驱动即时评估
- `task_evaluate_resonance()` 每 15min → SignalPipeline 数据到达即评分
